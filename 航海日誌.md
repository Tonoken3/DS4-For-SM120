### TP-6 OOM は **ctx起因**（重みではない）— 解消法判明
- 重み82GiBは6GPU(13.7GiB/GPU)に載る。OOMするのは最後の `PP GPU graph allocation`(活性/KV scratch)で、**ctxに比例**。`-c 256`はOOM、**`-c 64`は動作（正答 "Hello! How can I assist you today"）**、`-c 128`も動く(21.6)。
- → 実用ctxでTPを回すには活性/KV scratchの削減が必要（GPU2は表示GPU、配分から外すなら `CUDA_VISIBLE_DEVICES=0,1,3,4,5,6`）。

### ① 律速の切り分け結果（決定的）
PP_PROFILE 定常トークン内訳（51.7 ms/token = 19.3 t/s）: embed 0.015 / gpu0(7層) 7.81 / gpu1-6(各6層) 6.9–7.0 / p2p 各0.01(**無視可**) / output head 2.09。

- **発見1: 7GPUは完全直列**。バッチ1はレイヤ依存で並列不能 → PPは容量のためだけ。各GPUは自分の~7msだけ稼働し**86%アイドル**（SM実測14–18%＝デューティ比一致）。どのGPUも飽和せず。
- **発見2: MoEは軽い**。CUDAイベント計測で MoE GPU実計算 = **0.296 ms/層**（43層計12.73ms）＝チャンク壁の26%だけ。残り0.85ms/層は非MoEの小q8 matvec群（q/kv投影・compressor・indexer・output_proj・shared FFN）。
- **発見3（核心・仮説を覆す）: launch律速ではない。** `issue`（同期なしでカーネル発行のみ）= **0.4ms/chunk**、`gpu_ms`（発行+GPU完了待ち）= 7ms。ホストは0.4msで全発行を終え、残り6.6msはGPUが実行中。→ **GPU実行=メモリ帯域バウンド**（バッチ1で各層のq8/iq2重みを読む時間）。
- **結論: CUDA Graph化(②)は無効**（ホスト発行は全体でも2.8ms）。効くのは「読むバイトを減らす／並列に読む」＝**TPで重み行列を分割**。これがTPが正しい理由。

### 25 t/s への道（残差の正体）
TP_PROFILE: token あたり compute=43ms(94%) / all_reduce=2.3ms(5%) / combine=0.3ms(1%)。**all-reduce/同期は安い、computeが支配**。
- TP-6で既に**シャード済**: attention(head分割+O-proj partial+all_reduce)、MoE(expert分割)、**shared FFN(col/row分割+all_reduce, 確認済 ds4.c:10235)**。
- **まだ複製（両rankで全実行）= stem**: attn norm / q_a・q_b投影 / kv投影 / compressor(indexer_compressor_gate/kv) / indexer(attn_q_b + top-k) / RoPE。これがTPの床（43msの主因）。
- **次の一手 = stem のシャード**（正答性リスク大、O-proj同様の検証治具必須）:
  - q_b は head-range分割可（attentionが既にhead分割→各rankは自分のheadのqだけ必要、all-reduce不要）。最有力・比較的安全。
  - compressor/indexer は top-k が大域reductionで難。= ユーザの言う「③ A2式compressor削減」。
  - 見積り: stem半減で 43→~30ms、+output等で ~33ms → 30 t/s 圏（理想）。現実は要測定。
- 補助の安全策: output head(2.09ms, BW-bound)を最終stageの2rankでvocab分割 → ~1ms。+~0.6 t/s。

### このセッションでの変更（コミット対象）
- `ds4.c`: 計測器2つ（挙動不変・profile時のみ動作）:
  1. PP_PROFILE に `issue_ms`（同期なしの発行のみ時間）＝ ①発見3（host発行0.4ms vs GPU7ms）の計測器。
  2. TP_PROFILE に `barrier_sync`（per-layerバリアの累積時間）＝ B判定の計測器。結果: barrier_sync=34ms はほぼGPU実計算待ち＝Bの天井は低い（回収可能 ~3-5ms のPCIe all-reduceバブルのみ）。
- TP barrier ループは `TP_BARRIER(s,tp)` マクロに集約（中身は同一の per-rank ds4_gpu_synchronize、計測フック付き）。

### B/A の最終判断（このセッションの締め）
- **B（同期/all-reduce集約）= 小**（+1〜2 t/s）。barrier_sync 34ms は実計算待ちで無駄ではない。ただし12GPUでall-reduce多用時に効く土台。未実装。
- **A（stemシャード）= 本丸だが章規模**。stemは小matmul群（q_a 4.25MiB/kv 2.12MiB/compressor）＋大物 indexer。単発で+4t/sの安手は無い。
  - **indexer(attn_q_b 16MiB) head分割**が最大物。要: F16 col-matmul治具＋部分score all-reduce＋top-k複製＋検証治具。**indexerは大ctxでのみ発火**（c64では走らない）＝**12GPU長文脈で最大効果**。未着手。
- 推奨次手: GPU到着後、長ctx前提で indexer シャード章に着手。

## Multi-GPU MoE (Level 3)

### 設計

256 experts を 7 GPU に分割（各 ~37 experts）、各 GPU にプリロード。
Router で選択された 6 experts を担当 GPU にディスパッチ、並列計算、結果マージ。

### 実装経緯

#### バグ修正 (GPT 協力)
1. **CPU スタック配列を CUDA kernel に渡していた**: `local_idx`/`local_w` を device にコピーせずに kernel 引数として渡していた → illegal memory access
2. **grid 次元が過大**: kernel の row 次元 grid に `expert_mid_dim=5120` を指定 → 実際は `2048`。過大 grid がバッファ外書き込み
3. **ハードコード定数の誤り**: `DS4_N_EMBD=2048` と仮定 → 実は `4096`。`DS4_N_FF_EXP=5120` → 実は `2048`。tensor layout は (d0, d1, d2) で gate=(4096,2048,256), down=(2048,4096,256)。
   - gate: dim[0]=4096=DS4_N_EMBD, dim[1]=2048=DS4_N_FF_EXP
   - down: dim[0]=2048=DS4_N_FF_EXP, dim[1]=4096=DS4_N_EMBD

#### 解決済み
- ✅ `selected`/`weights` バッファを GPU scratch に追加
- ✅ CPU→GPU コピー後に kernel 起動
- ✅ grid 次元を正しい値に修正
- ✅ 全定数をモデル実値に合わせる
- ✅ 結果加算 (`moe_out[i] += partial[i]`)

#### 確認済み: `ds4 --backend hybrid` で正しい出力 ("Hello! How can I")

### 課題: 逐次 7 GPU 処理のオーバーヘッド

現在の mgpu は各 GPU を直列処理:
```
for g in 0..6:
    cudaSetDevice(g)
    cudaMemcpy H2D (input)
    quantize kernel
    gate/up kernel
    quantize mid
    down kernel
    cudaMemcpy D2H (result)
    accumulate
```

計 7× cudaSetDevice + 7× cudaMemcpy H2D + 7× cudaMemcpy D2H per layer → ~1.5ms/layer

### 30 t/s へのロードマップ

| フェーズ | 対策 | 推定速度 |
|---------|------|---------|
| **Phase A** | **7 GPU 並列処理** (pthread で同時起動) | **0.2ms/layer → 45 t/s** |
| **Phase B** | **Level 2: attention 全 GPU 化** (Q/KV proj, attn, out proj) | **0.5ms/layer → 30+ t/s** |
| Phase C | カーネル融合 (quantize+gate_up, quantize+down) | 微増 |

**Phase A**: 各 GPU を pthread で並列起動。main thread は全 thread 完了を待つ。7 GPU のレイテンシが隠蔽される。

**Phase B**: `ds4_gpu_matmul_q8_0_tensor` + `attention_decode_heads_tensor` + `ds4_gpu_rms_norm_weight_tensor` を device 0 で直列実行。CPU でやっていた ~2.5ms/layer を ~0.3ms に短縮。
